# How it works in a Zone
## Terms
### RRSET
![[DNSSEC101-3.png]]
- Used within DNSSEC
- Set of all resource records with the same name and same type
- DNSSEC allows us to validate integrity of recordsets within DNS. It does not validate based on individual records
- CNAME(Canonical Name) records point at other records
- A Record - IPv4 Record
- AAAA Record - IPv6 Record
- MX(Email Exchange) record

### RRSIG & Zone Signing Key (ZSK)

![[DNSSEC101-4.png]]
- Uses the concepts of digital signatures for encryption and verification of data
- <mark style="background: #FFF3A3A6;">RRSIG is generated by using the private ZSK to sign the RRSET</mark>. RRSIG is a digital signature of an RRSET

### DNSKEY
DNSKEY record stores public keys to verify all RRSIGS in the zone. <mark style="background: #FFF3A3A6;">It is the public part of the Zone Signing Key</mark>
![[Pasted image 20230305193403.png]]
- **DNSSEC resolver** takes in the RRSIG MX, the MX RRSET, and the DNSKEY to verify the MX RRSET

### Key Signing Key
![[Pasted image 20230305194513.png]]
- KSK
- Uses asymmetric encryption - so there is a public key and a private key
- Private KSK is used to create RRSIG from DNSKEY record to create RRSIG DNSKEY
- Parent zone links to the public KSK of a zone - creates a chain of trust

# Chain of Trust
## Delegation Signer Record(DS Record)
**NS Record** - Name server resource record
A **DS record** stores a hash of the child zone's KSK in this record in the parent zone. Adding this record adds that the parent trust the child's KSK
A DS record set can be validated with an RRSIG DS record

# Signing Ceremony
![[Pasted image 20230305223008.png]]
DNSSEC strengthens authentication in DNS using digital signatures based on public key cryptography. With DNSSEC, it's not DNS queries and responses themselves that are cryptographically signed, but rather DNS data itself is signed by the owner of the data.
