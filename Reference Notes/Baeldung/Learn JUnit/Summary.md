## Module 1 — Getting Started with JUnit

### JUnit Ecosystem
JUnit is a lightweight, widely-used unit-testing framework for Java. It integrates with many libraries and plugins, most notably **Mockito** for mocking. Its main alternative is **TestNG**, though JUnit has largely caught up in features and remains more popular.

### Writing Tests
Install JUnit via Maven by adding it to `pom.xml`. A JUnit BOM can be added to `<dependencyManagement>` to align versions across JUnit dependencies. In IntelliJ, tests can be generated by right-clicking a class or method → **Generate → Test**. Mark test methods with `@Test` and verify outputs using `assert*` methods.

### Test Structure — Given / When / Then
Each unit test should follow the **given-when-then** pattern:
- **Given** — set up the data needed
- **When** — execute the action under test
- **Then** — assert the expected outcome

Test cases should cover: the **happy path** (normal successful flow), **corner cases** (edge or failure cases), and **state changes** (e.g., verifying an ID is assigned after saving).

**Unit tests** test individual components in isolation; **integration tests** test multiple components working together.

**Inversion of Control (IoC)** is a design pattern where external sources initialize dependencies, simplifying how other classes consume them. **Mocking** creates fake implementations of dependencies to supply controlled test data.

### Tests and Project Configuration
Maven's **SureFire plugin** is included by default in Maven projects. Tests can be run via:
- `mvn surefire:test`
- `mvn test` (internally invokes SureFire)
- `mvn install` / `mvn package` (tests run before packaging)

The test command can be configured to specify which tests to run.

## Module 2 — JUnit Basics

### Lifecycle Methods
Lifecycle annotations allow setup and teardown logic to run around tests:

| Annotation | When it runs |
|---|---|
| `@BeforeAll` | Once before all tests in the class |
| `@BeforeEach` | Before each individual test |
| `@AfterEach` | After each individual test |
| `@AfterAll` | Once after all tests in the class |

`Before*` methods are used to initialize shared resources; `After*` methods clean them up (e.g., closing database connections or file readers).

### Mocking with Mockito
Mocking creates a fake implementation of a dependency. Mockito simplifies this significantly.

**Manual setup:**
```java
this.repository = Mockito.mock(CampaignRepository.class);
Mockito.when(repository.findById(1L)).thenReturn(Optional.empty());
```

**Annotation-based setup** (simpler):
```java
@ExtendWith(MockitoExtension.class)
class MyTest {
    @Mock
    private CampaignRepository repository;

    @InjectMocks
    private DefaultCampaignService service;
}
```
- `@ExtendWith(MockitoExtension.class)` — runs `MockitoAnnotations.openMocks()` automatically
- `@Mock` — creates a mock instance
- `@InjectMocks` — injects mocks into the target class

### JUnit Assertions
Assertions verify that an output matches expectations. Common methods include: `assertEquals`, `assertSame`, `assertNull`, `assertTrue`, `assertFalse`.

### Test Naming
- `@DisplayName` — sets a custom display name at the class or method level
- `@DisplayNameGenerator` — auto-generates display names from method names; custom generators can be implemented

### Disabling Tests
Use `@Disabled("reason")` to temporarily skip a test. Common reasons include: a dependency or external service being down, ongoing code changes, or a known issue being investigated.

## Module 3 — Advanced JUnit Assertions

### Expecting Exceptions
Use `assertThrows` to verify that a method throws the expected exception (unhappy path testing):
```java
assertThrows(SomeException.class, () -> service.handle());
```

### Hamcrest Assertions
Hamcrest is a third-party assertion library that produces more descriptive failure messages:
```java
assertThat(tasks, hasSize(2));
assertThat(tasks, containsInAnyOrder(task1, task2));
assertThat(tasks, everyItem(allOf(
    hasProperty("assignee", hasProperty("id", is(100L))),
    hasProperty("name", startsWith("BACKEND"))
)));
```

### AssertJ Assertions
AssertJ is another third-party library offering a fluent, chained assertion style with strong IDE autocompletion support:
```java
assertThat(tasks)
  .hasSize(2)
  .allMatch(task -> task.getAssignee().getId() == 100L)
  .noneMatch(task -> task.getName().contains("FRONTEND"))
  .containsExactlyInAnyOrder(task1, task2);
```

### Grouping Assertions with `assertAll`
`assertAll` runs all assertions in a block simultaneously, collecting and displaying all failures at once — unlike individual assertions that stop on the first failure:
```java
assertAll("Campaign assertions",
    () -> assertTrue(foundCampaign.isPresent()),
    () -> assertEquals("CA-123", foundCampaign.get().getCode()),
    () -> assertFalse(foundCampaign.get().isClosed())
);
```

## Module 4 — Advanced Test Types

### Repeated Tests
`@RepeatedTest(n)` runs a test `n` times. Useful for testing flaky behavior, idempotency, caching, or intermittent issues. Supports a custom display name and a `failureThreshold` to stop after a set number of failures.

### Parameterized Tests
`@ParameterizedTest` runs a single test method once per value provided by a `*Source` annotation:
- `@ValueSource` — a set of values
- `@NullSource` / `@EmptySource` — null or empty values
- `@EnumSource` — enum constants
- `@MethodSource` — values returned by a factory method
- `@CsvSource` / `@CsvFileSource` — comma-separated values inline or from a file

### Dynamic Tests
`@TestFactory` generates tests at runtime rather than compile time. The factory method must return a `Stream<DynamicTest>` or `Stream<DynamicNode>`. Use `DynamicContainer` to logically group multiple `DynamicNode` instances (both `DynamicTest` and `DynamicContainer` extend `DynamicNode`).

## Module 5 — Managing Tests with JUnit

### Execution Order
Test execution order is not guaranteed by default. Use `@TestMethodOrder` or `@TestClassOrder` (at the class level) to enforce ordering. Available ordering strategies via `MethodOrderer`:
- Display name
- Class name
- `@Order` annotation
- Random

### Tagging and Filtering
`@Tag("tag-name")` (method-level) marks tests with a label. Custom annotations can wrap `@Tag` for convenience. Filter tests at run time with:
```
mvn test -Dgroups=campaigns
```

### Test Suites
A **Test Suite** groups related tests together. Mark a class with `@Suite` and pair it with selection/exclusion annotations:
- `@SelectPackages`, `@SelectClasses`, `@SelectMethods` / `@SelectMethod`
- `@ExcludePackages`, `@IncludeTags`

### Conditional Test Execution
Tests can be enabled or disabled at runtime based on system conditions:
- `@EnabledOnOs` / `@EnabledOnJre`
- `@EnabledIfEnvironmentVariable` / `@DisabledIfSystemProperty`
- `@EnabledIf("methodName")` — uses a custom boolean method

### Parallel Execution
Enable parallel test execution in `src/test/resources/junit-platform.properties`:
```properties
junit.jupiter.execution.parallel.enabled=true
junit.jupiter.execution.parallel.mode.default=concurrent
```
Alternatively, annotate a class with `@Execution(ExecutionMode.CONCURRENT)`. Use `@ResourceLock("lock-name")` to prevent race conditions when multiple tests access the same shared resource.
